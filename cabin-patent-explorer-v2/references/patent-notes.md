# 智能座舱专利探索注意事项

## 一、2026年新规硬性要求

| 事项 | 要求 | 风险 |
|------|------|------|
| 文件格式 | 必须XML格式提交 | 非XML直接拒收 |
| 发明人 | 必须为自然人，写身份证号 | 禁止填"课题组"或"AI" |
| 同日申请 | 发明授权前必须放弃实用新型 | 不能同时保留两者 |
| 伦理合规 | 涉及人脸/生物识别的必须声明合规 | 违反直接驳回 |

## 二、专利三性核心要求

### 新颖性——前所未有
- 申请日前全球范围内无公开
- 警惕：展会、论文、内部技术分享都算公开
- 方案需检索确认无类似组合技术

### 创造性——非显而易见
- 2026年新规：仅场景转换不算创造性
- 必须有量化技术效果（误报率降低X%、成本降低X%）
- 算法必须结合硬件+场景，不能纯算法

### 实用性——可实施+有效果
- 说明书中必须记载具体实现方式
- 不能只写功能效果，需提供参数范围和实施路径

## 三、智能座舱专利特别注意事项

### 涉及AI/感知算法的专利
- 禁止"黑箱"写法：必须详细描述模型结构、训练步骤、参数范围
- 传感器融合类：需说明各传感器数据如何对齐、融合、仲裁
- DMS/OMS类：涉及生物识别，必须声明符合《个人信息保护法》

### 涉及多域联动的专利
- 座舱与智驾感知共享、OTA功能安全——需论证跨域协作的技术实现
- 系统架构类：微内核、虚拟化方案需符合ISO 26262功能安全

### 涉及用户数据的专利
- 数据采集必须合法
- 算法逻辑不得包含歧视性特征
- 高风险应用（疲劳告警、活体检测）需明确风险防控机制

## 四、权利要求书撰写策略

### 独立权利要求
- 采用"最小必要技术特征"原则
- 保护范围最大化，但需有实施支撑
- 框架适合作为独立权利要求的结构：条件A → 冲突B → 步骤C → 资源D → 问题E

### 从属权利要求
- 逐级细化，为审查意见提供退守方案
- 建议每个核心创新点准备3-5个替代实施例

## 五、避坑要点

| 坑 | 后果 | 应对 |
|----|------|------|
| 申请前公开（论文/展会/产品发布） | 新颖性丧失 | 先申请后公开 |
| 权利要求过宽（"高效""智能"等模糊词） | 不清楚被驳回 | 用具体技术特征替代 |
| 说明书公开不充分 | 驳回 | 详细记载实施例和参数 |
| 纯算法申请 | 客体不适格 | 必须结合硬件和场景 |
| 场景简单迁移 | 创造性不足 | 必须有量化技术效果 |

## 六、OEM买单的核心逻辑

能壁垒的核心：**别人抄你，绕不过去。**

OEM买专利的两个原因：
1. **法规合规**（不装不让卖）： DMS、儿童遗留检测、紧急呼叫
2. **降本**（减少硬件、减少调试成本）

**真正有技术壁垒的方案特征：**
- 硬件整合方案优于纯算法
- 需要大量实车数据训练，别人没有数据就是黑盒
- 涉及功能安全认证，门槛高
