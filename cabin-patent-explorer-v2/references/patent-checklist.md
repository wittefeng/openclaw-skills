# 专利审查自检清单

## 完整检查清单（10项）

### 1. 条件A：触发场景是否具体？

**检查要点**：
- [ ] 是否有明确的触发条件？
- [ ] 是否有量化阈值？（车速、时间、距离等）
- [ ] 是否区分不同场景？（高速/低速/泊车）

**合格示例**：
```
✅ 车速>60km/h且持续时间>30min
✅ 驾驶员闭眼时间>2秒且置信度>0.85
✅ 蓝牙钥匙距离车辆>10m且OMS检测到儿童存在
```

**不合格示例**：
```
❌ 当驾驶员疲劳时（无量化）
❌ 高速场景下（无阈值）
❌ 车辆运行时（太宽泛）
```

---

### 2. 冲突B：现有痛点是否量化？

**检查要点**：
- [ ] 是否明确现有技术的问题？
- [ ] 是否量化了问题的严重程度？（百分比/数值）
- [ ] 是否说明了问题的技术原因？

**合格示例**：
```
✅ 单一声音告警有效唤醒率<40%，误报率>15%
✅ OMS+DMS双系统硬件成本增加¥800，功耗增加15W
✅ 泊车场景DMS误报率>60%，算力浪费30%
```

**不合格示例**：
```
❌ 现有方案效果不好（无量化）
❌ 用户不满意（主观描述）
❌ 成本较高（无具体数值）
```

---

### 3. 步骤C：关键步骤是否具体？

**检查要点**：
- [ ] 是否描述了算法的关键步骤？
- [ ] 是否避免了"黑箱"描述？（必须说明模型结构/训练步骤）
- [ ] 是否有明确的逻辑顺序？

**合格示例**：
```
✅ 步骤1：疲劳置信度多级判定（≥0.7/≥0.85/≥0.95）
✅ 步骤2：根据置信度分级触发不同干预策略
✅ 步骤3：振动→空调→转向的时序配合（间隔200ms）
```

**不合格示例**：
```
❌ 使用AI算法检测疲劳（黑箱）
❌ 系统智能判断并干预（无步骤）
❌ 通过深度学习模型处理（无结构说明）
```

---

### 4. 资源D：硬件参数是否明确？

**检查要点**：
- [ ] 是否指定了硬件组件？
- [ ] 是否有具体的参数范围？（频率/响应时间/功耗）
- [ ] 是否说明了硬件之间的连接关系？

**合格示例**：
```
✅ 座椅振动执行器：频率可调8-15Hz，响应时间<100ms
✅ 空调风门步进电机：响应时间<200ms，精度±1°
✅ OMS摄像头：低功耗模式5fps，功耗<50mW
```

**不合格示例**：
```
❌ 座椅振动器（无参数）
❌ 空调系统（不具体）
❌ 摄像头（无型号/参数）
```

---

### 5. 效果F：技术效果是否量化？

**检查要点**：
- [ ] 是否有具体的数值指标？（百分比/绝对值）
- [ ] 是否与现有技术有对比？
- [ ] 效果是否可验证？（测试方法/数据来源）

**合格示例**：
```
✅ 有效唤醒率从40%提升至85%（提升112.5%）
✅ 误报率从15%降至3%（降低80%）
✅ BOM成本降低30%（从¥1200降至¥840）
```

**不合格示例**：
```
❌ 提升了用户体验（无量化）
❌ 效果更好（无对比）
❌ 大幅降低成本（无具体数值）
```

**2026年硬性要求**：必须有量化技术效果，否则直接驳回！

---

### 6. 对比G：与现有技术的对比是否清晰？

**检查要点**：
- [ ] 是否明确现有技术方案？
- [ ] 是否有量化对比数据？
- [ ] 是否说明了提升的原因？（技术原理）

**合格示例**：
```
✅ 相比单一声音告警，有效唤醒率从40%提升至85%（+112.5%）
   原因：多模态干预（振动+空调+转向）比单一声音更有效
✅ 相比DMS+OMS双系统，BOM成本降低30%（-¥360）
   原因：单目广角摄像头复用，减少1颗摄像头和1颗SOC
```

**不合格示例**：
```
❌ 比现有技术更好（无具体对比）
❌ 有显著提升（无量化）
❌ 优于传统方案（未说明传统方案是什么）
```

---

### 7. 壁垒H：技术壁垒是否论证充分？

**检查要点**：
- [ ] 是否说明了为什么别人绕不过去？
- [ ] 是否分析了技术难点？（多域协同/时序配合/参数匹配）
- [ ] 是否有数据支撑壁垒强度？

**合格示例**：
```
✅ 需要座椅振动执行器（8-15Hz可调）+ 空调风门电机（<200ms响应）
   + 多级置信度判定算法的协同控制
   关键难点：振动-空调-转向的时序配合（振动先于空调200ms启动，
   转向请求需双重确认）和参数匹配（不同车速对应不同振动频率）
   单一技术无法实现85%的有效唤醒率
```

**不合格示例**：
```
❌ 技术难度高（无具体说明）
❌ 有创新性（未论证壁垒）
❌ 别人很难复制（主观判断）
```

---

### 8. 新颖性：是否经过检索确认？

**检查要点**：
- [ ] 是否进行了专利检索？（Google Patents + CNABS）
- [ ] 是否分析了最接近现有技术？
- [ ] 是否论证了与现有技术的差异？

**检查项**：
```
□ 检索了Google Patents（快速初筛）
□ 检索了CNABS（中国专利深度检索）
□ 检索了DWPI（海外专利）
□ 分析了3-5篇最接近专利
□ 确认了技术方案的组合新颖性
□ 未发现相同技术方案
```

**风险警示**：
- 申请前公开（论文/展会/产品发布）→ 新颖性丧失
- 检索不充分 → 被现有技术驳回

---

### 9. 创造性：是否论证非显而易见性？

**检查要点**：
- [ ] 不是简单场景迁移？（2026年新规）
- [ ] 是否有量化技术效果支撑创造性？
- [ ] 是否说明了技术启示的缺乏？

**创造性论证模板**：
```
现有技术1（DMS疲劳检测）+ 现有技术2（座椅振动）
≠ 本方案（多级置信度+时序配合+参数匹配）

理由：
1. 现有技术1只检测疲劳，未涉及干预策略
2. 现有技术2只提供振动反馈，未与疲劳检测联动
3. 本方案的关键在于"多级置信度判定+多执行器时序配合"，
   这不是简单组合，而是有协同效应的技术方案
4. 量化效果证明：有效唤醒率从40%→85%，简单组合无法达到
```

**2026年新规**：仅场景转换不算创造性！必须有量化技术效果。

---

### 10. 实用性：是否有具体实施路径？

**检查要点**：
- [ ] 是否有具体的参数范围？（可实施）
- [ ] 是否有硬件选型建议？（可采购）
- [ ] 是否有软件实现逻辑？（可编程）

**合格示例**：
```
✅ 座椅振动执行器选型：型号XXX，频率范围8-15Hz，功率20W
✅ 置信度判定算法：基于ResNet-18，输入112×112图像，
   输出3级置信度（0.7/0.85/0.95阈值）
✅ 时序控制逻辑：振动启动→200ms后空调启动→500ms后评估效果
```

**不合格示例**：
```
❌ 使用合适的振动器（不具体）
❌ 根据情况调整参数（无范围）
❌ 算法自动处理（无逻辑）
```

## 合规性检查

### 伦理合规（涉及生物识别）

```
□ 数据采集合法性声明（符合《个人信息保护法》）
□ 算法无歧视性特征说明
□ 用户知情同意机制
□ 数据安全保护措施
□ 高风险应用（疲劳检测、活体检测）风险防控机制
```

### 功能安全（涉及DMS/eCall/OTA）

```
□ 符合ISO 26262功能安全要求
□ 明确ASIL等级（ASIL B/C/D）
□ 安全机制说明（看门狗/冗余/故障自隔离）
□ 验证方法（HARA分析/FMEA/FTA）
```

## 常见驳回理由及应对

| 驳回理由 | 原因 | 应对策略 |
|---------|------|---------|
| 创造性不足 | 场景简单迁移/无量化效果 | 补充量化对比数据 |
| 不清楚 | 权利要求过宽/模糊词 | 用具体技术特征替代 |
| 公开不充分 | 缺少实施例/参数 | 补充详细实施例 |
| 纯算法申请 | 未结合硬件和场景 | 增加硬件执行器和场景描述 |
| 新颖性丧失 | 申请前已公开 | 无法挽救，需调整方向 |
