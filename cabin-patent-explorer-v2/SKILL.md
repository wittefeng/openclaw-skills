# 智能座舱专利探索技能 V2

## 核心增强功能

### 1. 量化效果自动生成器（新增）

**输入**：原始框架（条件A→冲突B→步骤C→资源D→问题E）
**输出**：完整补强版本（+效果F+对比G+壁垒H）

**自动生成规则**：

| 技术领域 | 量化指标模板 | 典型数值范围 |
|---------|------------|-------------|
| 感知算法 | 准确率/召回率/F1提升 | +15%~+40% |
| 系统架构 | 故障恢复时间/可用性 | <200ms / 99.99% |
| 硬件整合 | BOM成本降低/功耗降低 | -20%~-50% |
| 用户体验 | 唤醒率/满意度/NPS | +30%~+100% |
| 功能安全 | ASIL等级/故障覆盖率 | ASIL B→D / >95% |

**示例转换**：

```
原始：解决了疲劳告警"叫不醒人"的问题

补强后：
- 效果F：有效唤醒率从40%提升至85%
- 对比G：相比单一声音告警提升112.5%，误触发率从15%降至3%
- 壁垒H：需要座椅振动执行器（8-15Hz可调）+ 空调风门电机（<200ms响应）
         + 多级置信度判定算法的协同控制，单一技术无法复制效果
```

### 2. 检索指引（新增）

**推荐数据库**：
- CNABS（中国专利全文数据库）
- DWPI（德温特世界专利索引）
- Google Patents（免费快速检索）
- Incopat（语义检索）

**检索策略模板**：
```
关键词组合 = 核心技术词 + 应用场景 + 解决手段
示例：
- 核心技术：DMS、疲劳检测、置信度、多级判定
- 应用场景：高速巡航、长时间驾驶、疲劳干预
- 解决手段：座椅振动、空调联动、分级告警

检索式：(DMS OR 疲劳检测) AND (置信度 OR 多级) AND (振动 OR 联动)
```

### 3. 审查自检清单（新增）

```
□ 条件A：是否有具体触发阈值？（如：车速>60km/h且持续>30min）
□ 冲突B：是否量化了现有痛点？（如：误报率>60%，唤醒率<40%）
□ 步骤C：是否描述了关键算法步骤？（非黑箱描述）
□ 资源D：是否指定了硬件参数？（如：振动频率8-15Hz，响应<200ms）
□ 效果F：是否有量化技术效果？（百分比/数值提升）
□ 对比G：是否与现有技术有量化对比？
□ 壁垒H：是否论证了技术壁垒？（为什么别人绕不过去）
□ 新颖性：检索确认无类似组合？
□ 创造性：有非显而易见的论证？（不是简单场景迁移）
□ 实用性：有具体实施路径？（参数范围、硬件选型）
□ 合规性：涉及生物识别是否声明伦理合规？
```

### 4. OEM价值评估（新增）

**价值维度评分**：

| 维度 | 权重 | 评分标准 | 示例 |
|------|------|---------|------|
| 法规合规 | 40% | 是否强制要求 | DMS法规=10分 |
| 降本价值 | 30% | BOM降低幅度 | -30%=9分 |
| 体验提升 | 20% | 用户满意度提升 | +40%=8分 |
| 技术壁垒 | 10% | 复制难度 | 需多域协同=9分 |

**综合评分**：
- ≥8.5分：强烈推荐申请，OEM买单意愿强
- 7-8.5分：建议申请，需补强壁垒论证
- <7分：谨慎申请，可能纯算法或壁垒不足

## 工作流程 V2

### Step 1: 需求确认
询问或确认：
- **应用场景**：自用/产品化/卖给OEM？
- **技术领域**：感知算法/系统架构/硬件整合/用户体验？
- **时间要求**：快速产出（1周）/高质量（1月）？

### Step 2: 方向生成
使用框架生成器：
```
由于[条件A]发生了[冲突B]，
系统通过[步骤C]调用了[资源D]，
最终解决了[问题E]，
实现了[效果F]，
相比现有技术[对比G]，
技术壁垒在于[壁垒H]
```

### Step 3: 自动补强
调用量化效果生成器，自动填充：
- 效果F：基于技术领域匹配量化指标模板
- 对比G：计算与现有技术的提升百分比
- 壁垒H：分析资源D的协同复杂度

### Step 4: 检索验证
提供检索指引，确认新颖性

### Step 5: 审查自检
对照清单逐项检查

### Step 6: OEM价值评估
输出综合评分和建议

### Step 7: 输出完整方案
包含：
1. 专利方向概述
2. 完整框架（A-H）
3. 检索报告
4. 审查自检结果
5. OEM价值评估
6. 权利要求书草稿

**用户输入**：
> "我想申请一个关于座舱内儿童安全监控的专利，能检测儿童是否被遗忘在车内"

**系统输出**：

```
📋 专利方向：儿童遗留检测与多级干预系统

🎯 完整框架：
由于[车辆熄火后座舱OMS摄像头检测到儿童存在且驾驶员已离开]
发生了[现有方案只有声音提醒，驾驶员已走远无法听到，儿童长时间滞留导致窒息风险]
系统通过[儿童存在置信度分级 + 驾驶员距离检测 + 干预策略联动]

最终解决了[驾驶员远离场景下单一声音提醒无效]的问题

📊 OEM价值评估：
- 法规合规：10分（欧盟已强制要求，国内法规跟进中）

- 技术壁垒：8分（跨域协同+低功耗算法）
```

## 注意事项

1. **量化效果是硬性要求**：2026年审查必须有具体数值，不能只有"提升""改善"
2. **自动生成的数值需验证**：基于行业典型值，实际申请需实车数据支撑
3. **检索是必要步骤**：框架不能替代检索，必须确认新颖性
4. **壁垒论证是关键**：OEM买单的核心是"别人绕不过去"

## 重要使用规范

### 主动探索模式（默认）

**当用户说"帮我探索新的专利方向"时，系统应**：

1. **不等待用户指定方向**，主动基于技术空白点生成新方向
2. **分析现有10个方向库**，识别未被覆盖的技术领域
3. **生成3-5个新专利方向**，每个包含简要框架和价值评估
4. **由用户选择**深入哪个方向，而非让用户先指定

**禁止行为**：
- ❌ 问用户"你想探索哪个方向？"
- ❌ 让用户从预设选项中选择
- ❌ 等待用户提供技术场景

**正确行为**：
- ✅ 主动分析技术空白
- ✅ 生成新方向供用户选择
- ✅ 基于空白领域创造方向

### 技术空白识别方法

```
Step 1: 分析现有方向库的技术领域覆盖
   - 感知融合：DMS/OMS/手势/仲裁 ✅
   - 系统架构：OTA/故障隔离 ✅
   - 音频交互：隐私隔离 ✅
   - 用户体验：疲劳干预/遗忘提醒 ✅
   - 成本优化：硬件复用 ✅

Step 2: 识别未覆盖的技术空白
   - 情绪感知、健康监测、宠物检测（感知类）
   - 能耗管理、热管理、安全防护（架构类）
   - 主动降噪、声场定位、个性化音效（音频类）
   - 儿童娱乐、氛围营造、健康关怀（体验类）
   - 算力共享、能源优化（成本类）

Step 3: 基于空白生成新方向
   - 每个新方向需符合A→H框架
   - 提供量化效果预测
   - 进行OEM价值预评估
```

### 新方向生成标准

每个新方向必须包含：
1. **一句话描述**：技术创新的核心
2. **A→H完整框架**：条件→冲突→步骤→资源→问题→效果→对比→壁垒
3. **量化效果预测**：基于技术领域的典型值
4. **OEM价值预评估**：4维度评分
5. **推荐优先级**：P0/P1/P2分级
