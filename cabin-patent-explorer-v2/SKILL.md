# 智能座舱专利探索技能 V2

## 快速开始

### 触发方式

```
用户输入 → 系统识别 → 自动加载技能 → 执行
```

**触发关键词：**
- "帮我探索专利"
- "生成专利框架"
- "评估专利价值"
- "写权利要求书"
- "检查新颖性"
- "智能座舱专利"

### 工作流程

```
Step 1: 查重检查（CRITICAL）
   └─ 读取专利存档（references/patent-archive.md）
   └─ 确认方向不在"已做"列表
   └─ 执行查重检索（Google Patents + CNABS + 车企资料）
   └─ 确认技术差异点

Step 2: 需求确认
   └─ 确定应用场景（自用/产品化/OEM）
   └─ 确定技术领域（感知/架构/硬件/体验）
   └─ 确定时间要求（快速/高质量）

Step 3: 方向生成
   └─ 使用A→H框架生成专利方向
   └─ 自动匹配量化指标

Step 4: 自动补强
   └─ 生成效果F（量化指标）
   └─ 生成对比G（与现有技术对比）
   └─ 生成壁垒H（技术壁垒论证）

Step 5: 检索验证
   └─ 提供检索策略
   └─ 分析现有专利
   └─ 更新专利存档（如发现有"已做"方向）

Step 6: 审查自检
   └─ 对照10项检查清单
   └─ 标记风险点

Step 7: OEM价值评估
   └─ 4维度评分
   └─ 综合评分与建议

Step 8: 输出完整方案
   └─ 查重结果摘要
   └─ 专利方向概述
   └─ 完整框架（A-H）
   └─ 检索报告
   └─ 审查清单
   └─ OEM评估
   └─ 权利要求书草稿
   └─ 更新专利存档
```

## 核心功能

### 1. 量化效果自动生成器

**输入：** 原始框架（条件A→冲突B→步骤C→资源D→问题E）
**输出：** 完整补强版本（+效果F+对比G+壁垒H）

**自动生成规则：**

| 技术领域 | 量化指标模板 | 典型数值范围 |
|---------|------------|-------------|
| 感知算法 | 准确率/召回率/F1提升 | +15%~+40% |
| 系统架构 | 故障恢复时间/可用性 | <200ms / 99.99% |
| 硬件整合 | BOM成本降低/功耗降低 | -20%~-50% |
| 用户体验 | 唤醒率/满意度/NPS | +30%~+100% |
| 功能安全 | ASIL等级/故障覆盖率 | ASIL B→D / >95% |

**示例转换：**

```
原始：解决了疲劳告警"叫不醒人"的问题

补强后：
- 效果F：有效唤醒率从40%提升至85%
- 对比G：相比单一声音告警提升112.5%，误触发率从15%降至3%
- 壁垒H：需要座椅振动执行器（8-15Hz可调）+ 空调风门电机（<200ms响应）
         + 多级置信度判定算法的协同控制，单一技术无法复制效果
```

### 2. 检索指引

**推荐数据库：**
- CNABS（中国专利全文数据库）
- DWPI（德温特世界专利索引）
- Google Patents（免费快速检索）
- Incopat（语义检索）

**检索策略模板：**
```
关键词组合 = 核心技术词 + 应用场景 + 解决手段

示例：
- 核心技术：DMS、疲劳检测、置信度、多级判定
- 应用场景：高速巡航、长时间驾驶、疲劳干预
- 解决手段：座椅振动、空调联动、分级告警

检索式：(DMS OR 疲劳检测) AND (置信度 OR 多级) AND (振动 OR 联动)
```

### 3. 审查自检清单

```
□ 条件A：是否有具体触发阈值？（如：车速>60km/h且持续>30min）
□ 冲突B：是否量化了现有痛点？（如：误报率>60%，唤醒率<40%）
□ 步骤C：是否描述了关键算法步骤？（非黑箱描述）
□ 资源D：是否指定了硬件参数？（如：振动频率8-15Hz，响应<200ms）
□ 效果F：是否有量化技术效果？（百分比/数值提升）
□ 对比G：是否与现有技术有量化对比？
□ 壁垒H：是否论证了技术壁垒？（为什么别人绕不过去）
□ 新颖性：检索确认无类似组合？
□ 创造性：有非显而易见的论证？（不是简单场景迁移）
□ 实用性：有具体实施路径？（参数范围、硬件选型）
□ 合规性：涉及生物识别是否声明伦理合规？
```

### 4. OEM价值评估

**价值维度评分：**

| 维度 | 权重 | 评分标准 | 示例 |
|------|------|---------|------|
| 法规合规 | 40% | 是否强制要求 | DMS法规=10分 |
| 降本价值 | 30% | BOM降低幅度 | -30%=9分 |
| 体验提升 | 20% | 用户满意度提升 | +40%=8分 |
| 技术壁垒 | 10% | 复制难度 | 需多域协同=9分 |

**综合评分：**
- ≥8.5分：强烈推荐申请，OEM买单意愿强
- 7-8.5分：建议申请，需补强壁垒论证
- <7分：谨慎申请，可能纯算法或壁垒不足

## 查重机制（CRITICAL）

### 查重流程（每次探索前必须执行）

**Step 1: 读取存档**
```
必须读取：references/patent-archive.md
确认用户描述的方向不在"已确认车企已做"列表
```

**Step 2: 执行检索**
```
数据库：
- Google Patents（全球专利）
- CNABS（中国专利）
- 车企公开资料（特斯拉/奔驰/宝马/蔚来/小鹏/理想）
- 供应商方案（博世/大陆/采埃孚/华为）

检索式：
(核心技术词) AND (座舱 OR 车内 OR 车辆) AND (应用关键词)
```

**Step 3: 确认差异**
```
□ 技术路线是否相同？
□ 应用场景是否相同？
□ 解决效果是否相同？
□ 是否有明显差异化空间？
```

**Step 4: 更新存档**
```
如发现新的"已做"方向，立即更新 patent-archive.md
记录：方向名称、技术描述、确认来源、日期
```

### 查重通过标准

- ✅ **未发现相同技术** → 继续生成专利方案
- ⚠️ **发现相似技术** → 标记差异点，评估差异化可行性
- ❌ **发现相同技术** → 放弃该方向，向用户说明并推荐替代方案

### 禁止行为

❌ **严禁**：
- 不查重直接推荐方向
- 已知"已做"仍推荐
- 不记录新发现的"已做"方向

✅ **必须**：
- 每次探索前查重
- 明确告知用户查重结果
- 及时更新存档

## 注意事项

1. **量化效果是硬性要求**：2026年审查必须有具体数值，不能只有"提升""改善"
2. **自动生成的数值需验证**：基于行业典型值，实际申请需实车数据支撑
3. **检索是必要步骤**：框架不能替代检索，必须确认新颖性
4. **壁垒论证是关键**：OEM买单的核心是"别人绕不过去"
5. **查重是底线**：不查重不推荐，已做方向绝不重复推荐
